# 历史、待看与无标签一致性修复设计

## 目标

修复待看进度与自动移除、历史页右键菜单、历史页删除缺档归档提示、无标签缺档元数据报错，以及取消无标签后未恢复全部归档。保持现有视觉、分页、筛选、Worker/LRR 同步和删除确认行为。

## 已确认根因

1. 待看记录只保存归档 ID 与加入时间。主页和待看页只监听 `lrr:watchlist-changed`，未合并或监听阅读历史进度，因此卡片不更新；超过 80% 自动移除又只在 Reader 当前页 effect 中执行，无法覆盖历史、Worker 或 LRR 水合得到的高进度。
2. `HistoryPage` 未接入现有 `ArchiveContextMenu`，只配置了长按删除历史。
3. 历史页没有右键删除归档链路，也没有处理“删除时归档已不存在”的 400/404 分支。
4. 官方 API 先返回无标签 ID，再逐个请求元数据。两次请求之间归档可能被删除；单个元数据 400/404 当前会拒绝整个批次。
5. 进入无标签会清空列表。快速取消时，普通列表请求可能命中 2.5 秒去重键而被跳过，空列表无法恢复。

## 方案

### 待看进度与自动移除

- 在现有待看模块增加小型纯函数：按归档 ID 将历史最高页合并到待看项，并计算严格超过 80% 的归档 ID。
- 主页已有历史状态，直接派生带进度的待看项；待看页增加历史状态并监听 `lrr:history-changed`。
- 两个待看视图都给 `ArchiveCard` 传入当前页和进度条配置。
- 历史进度变化后，批量调用现有 `removeWatchlistItems` 清理超过阈值的项。删除函数本身幂等，Reader 原有即时删除继续保留。
- 页数未知或无效时不自动删除，避免误删。

### 历史页右键菜单与删除归档

- 复用现有 `ArchiveContextMenu`。右键和移动端长按均打开标准菜单；保留阅读、编辑元数据、下载、复制链接、删除历史记录与删除归档。
- “删除归档”继续使用现有主题确认弹窗和 `deleteArchiveWithFavoriteSync`，不改变不可逆操作确认与 E-Hentai 收藏同步规则。
- 删除成功后立即清理对应历史和待看记录，再刷新页面状态。
- 若删除 API 返回 400/404，视为归档已不在 LRR：直接清理对应历史记录，关闭删除确认框，显示非破坏性单按钮主题弹窗：“归档已不存在于 LANraragi，相关历史记录已清理。”不保留该历史记录，不再次确认。
- 401、网络、5xx 等其他错误不清历史，使用主题弹窗显示错误。

### 无标签容错与取消恢复

- 给现有有界批量元数据加载器增加可选的“忽略缺档”策略，仅跳过 status 400/404，并保持其余结果顺序。
- Abort、401、网络和 5xx 继续抛出，由现有错误区展示。
- 无标签加载使用该策略；不修改官方 ID 列表接口与并发上限。
- 切换或取消无标签时使上一次请求去重键失效，让现有单一 fetch effect 必定按新分类状态请求；仍由 AbortController 与请求序号阻止旧响应提交。

## 测试

- 扩展无标签 Node 检查：默认批量失败语义不变；启用策略时跳过 400/404、保持顺序；非缺档错误与 Abort 仍拒绝。
- 新增小型 Node 检查：待看与历史按 ID 合并最高页；80% 不删除，超过 80% 删除；未知总页数不删除。
- 静态回归检查：主页和待看页订阅/使用历史进度；历史页接入共享右键菜单、主题确认框与缺档清理分支；无标签切换使去重键失效。
- 执行全部 `scripts/check-*.mjs`、生产构建和 `git diff --check`。

## 非目标

- 不建立全局状态库。
- 不改变 80% 阈值或“严格超过”语义。
- 不修改 LANraragi 数据库，也不主动清理服务器无标签 ID 列表。
- 不改变卡片尺寸、主题和动画。
